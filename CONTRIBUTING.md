# Contributing to Security Writeups

Thank you for your interest in contributing to this security research repository!

## Writeup Format

All writeups must follow this structure:

### Required Sections

1. **Title**: Clear, descriptive vulnerability name
2. **Overview**: Brief description, severity rating, OWASP/CWE references
3. **Technical Explanation**: How the vulnerability works
4. **Attack Scenario**: Real-world exploitation context
5. **Proof of Concept**: Working exploit examples
6. **Impact**: Consequences of successful exploitation
7. **Mitigation**: Defense strategies and secure coding practices
8. **Secure Code Example**: Complete, working secure implementation
9. **Security Takeaways**: Key lessons (10 points)
10. **References**: Links to authoritative sources

## Code Examples

- Use Python, JavaScript, Solidity, or other widely-used languages
- Include complete, runnable examples
- Add comments explaining security-critical sections
- Show both vulnerable and secure implementations

## Quality Standards

- Professional tone suitable for security researchers
- Technically accurate information
- Practical, actionable advice
- Well-formatted code with proper syntax highlighting
- Clear explanations accessible to intermediate developers

## Submission Process

1. Fork the repository
2. Create a new branch for your writeup
3. Follow the established naming convention: `##-vulnerability-name.md`
4. Ensure all sections are complete
5. Test code examples for accuracy
6. Submit a pull request with a clear description

## Review Criteria

- Technical accuracy
- Completeness of coverage
- Code quality and security
- Clarity of explanations
- Proper attribution and references

## Ethical Guidelines

- All content must be for educational purposes only
- Do not include exploits for unpatched vulnerabilities
- Respect responsible disclosure practices
- Do not target specific organizations without permission

## Questions?

Open an issue for discussion before starting major contributions.
